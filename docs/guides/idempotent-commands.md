# Idempotent Commands Guide

## Goal

Every command in SMScore is idempotent. Re-issuing the same command
(with the same idempotency key) produces the same result without
duplicating state changes.

## Why Idempotency

- **Network retries**: a client that times out and retries must not
  double-charge a student.
- **Offline replay**: queued offline commands are re-issued on
  reconnect. They must not duplicate.
- **Bulk operations**: a command that targets 1000 students and
  fails halfway must be replayable without creating duplicates.

## Idempotency Key

Every command carries an `IdempotencyKey`:

```rust
pub struct IdempotencyKey(Uuid);
```

The key is generated by the engine when the command is constructed.
For client-driven commands, the client may supply the key (e.g. for
a deterministic UUID based on the request body).

The engine stores `(idempotency_key, command_outcome)` in a dedup
table. On replay, the engine returns the stored outcome without
re-applying the command.

## Scopes

The dedup table is scoped to:

- `SchoolId` (commands are isolated per school).
- `CommandType` (different command types can have the same key).
- Optional `ResourceId` (the aggregate the command targets).

```sql
CREATE TABLE command_idempotency (
    idempotency_key UUID NOT NULL,
    school_id INT NOT NULL,
    command_type TEXT NOT NULL,
    resource_id UUID,
    outcome JSONB NOT NULL,
    recorded_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    PRIMARY KEY (idempotency_key, school_id, command_type)
);
```

## Storage Adapter

The storage adapter exposes:

```rust
#[async_trait]
pub trait IdempotencyStore: Send + Sync {
    async fn lookup(&self, key: IdempotencyKey, command: &str) -> Result<Option<CommandOutcome>>;
    async fn record(&self, key: IdempotencyKey, command: &str, outcome: &CommandOutcome) -> Result<()>;
}
```

The engine wraps the command in a transaction that:

1. Looks up the key.
2. If found, returns the stored outcome.
3. Otherwise, applies the command and records the outcome.

## Command Outcomes

The outcome is a serializable representation of the result:

```rust
pub struct CommandOutcome {
    pub status: OutcomeStatus,
    pub payload: serde_json::Value,
    pub events: Vec<EventId>,
}

pub enum OutcomeStatus {
    Success,
    Failure { error: String, code: &'static str },
}
```

The payload allows the caller to retrieve the same value object
(e.g. the created student) on replay.

## Idempotency Window

The dedup table retains records for a configurable window (default
**30 days**). After that, the engine may forget the key and treat a
replay as a fresh command. Consumers can extend the window for
compliance (e.g. 7 years for financial commands).

## Non-Idempotent Operations

Some operations cannot be made idempotent without a key:

- **Time-based events**: "send a notification at 9am" is not
  idempotent without an explicit "send" command.
- **External side effects**: a payment gateway call cannot be
  reversed by the engine. The payment port's idempotency key handles
  this.

For these, the engine relies on the underlying port's idempotency
(e.g. the payment port's `IdempotencyKey`).

## Worked Example

A command is constructed and dispatched:

```rust
let key = IdempotencyKey::new();
let cmd = AdmitStudentCommand {
    tenant,
    admission_no: ...,
    ...
};

let outcome = engine.dispatch_with_idempotency(key, cmd).await?;
// outcome: CommandOutcome { status: Success, payload: <student>, events: [...] }

// Replay with the same key returns the same outcome.
let outcome2 = engine.dispatch_with_idempotency(key, cmd).await?;
assert_eq!(outcome.payload, outcome2.payload);
```

A different key creates a new student:

```rust
let outcome3 = engine.dispatch_with_idempotency(IdempotencyKey::new(), cmd).await?;
// outcome3: Conflict (duplicate admission number) or different student
// (depending on business rule)
```

## Testing

- A test that re-issuing a command with the same key returns the
  same outcome.
- A test that re-issuing with a different key produces a conflict.
- A test of the dedup table retention.
- A test of cross-tenant key isolation.

## Audit

Every command outcome (success or failure) is recorded. The audit log
links to the dedup record via the idempotency key.
