# Storage Adapter Implementation Guide

## Goal

Build a working storage adapter for PostgreSQL and SQLite that
satisfies the storage port and the parity test suite.

## Adapter Skeleton

```rust
pub struct PostgresStorage {
    pool: sqlx::Pool<Postgres>,
}

impl PostgresStorage {
    pub async fn connect(url: &str) -> Result<Self> {
        let pool = sqlx::postgres::Pool::connect(url).await?;
        Ok(Self { pool })
    }
}

#[async_trait]
impl StorageAdapter for PostgresStorage {
    async fn begin(&self) -> Result<Transaction> {
        let tx = self.pool.begin().await?;
        Ok(Transaction::Postgres(tx))
    }
    // ...
}
```

## Repository Implementation

Each repository is a translator from the macro-emitted
`QueryNode<StudentField>` AST into a storage-specific execution plan.
The adapter **never** constructs field enums or relation enums; it
only pattern-matches on the variants it receives from the builder.

```rust
pub struct PostgresStudentRepository {
    pool: PgPool,
}

#[async_trait]
impl StudentRepository for PostgresStudentRepository {
    async fn get(&self, id: StudentId) -> Result<Option<Student>> {
        let row = sqlx::query!(
            "SELECT id, school_id, status, last_name, class_id, parent_id
             FROM students WHERE id = $1 AND school_id = $2",
            id.inner(),
            id.school().inner(),
        )
        .fetch_optional(&self.pool)
        .await?;
        row.map(Student::from_row).transpose()
    }

    async fn query(&self, q: StudentQuery) -> Result<Vec<Student>> {
        // The adapter consumes the AST that the macro-generated
        // StudentQueryBuilder emitted at await time. The macro
        // produced StudentField, StudentQueryBuilder, StudentRelation,
        // and StudentQuery. The adapter owns none of those types; it
        // only reads them.
        let plan = self.translate(&q);
        plan.execute(&self.pool).await
    }

    // ...
}
```

The `StudentField` enum is generated by the `#[derive(DomainQuery)]`
macro. The adapter never constructs it; it only pattern-matches on
the variant it receives.

## Query Translation

The macro emits the AST; the adapter emits the SQL. This separation is
explicit. The translation step walks the `QueryNode<StudentField>` AST
and emits parameterized SQL. Because the AST is closed and finite, the
`match` is exhaustive at compile time — adding a new operator requires
a new arm or the build fails.

```rust
impl PostgresStudentRepository {
    fn translate(&self, q: &StudentQuery) -> SqlPlan {
        let mut sql = String::from(
            "SELECT id, school_id, status, last_name, class_id, parent_id
             FROM students WHERE school_id = $1"
        );
        let mut params: Vec<Box<dyn ToSql>> = vec![Box::new(q.school_id().inner())];

        for node in q.filters() {
            self.emit_node(node, &mut sql, &mut params);
        }

        for order in q.orders() {
            sql.push_str(&format!(
                " ORDER BY {} {}",
                order.field.column(),
                order.direction.as_sql()
            ));
        }

        sql.push_str(&format!(" LIMIT {} OFFSET {}", q.limit(), q.offset()));
        SqlPlan { sql, params }
    }

    fn emit_node(
        &self,
        node: &QueryNode<StudentField>,
        sql: &mut String,
        params: &mut Vec<Box<dyn ToSql>>,
    ) {
        let placeholder = params.len() + 1;
        match node {
            QueryNode::Eq(field, value) => {
                sql.push_str(&format!(" AND {} = ${}", field.column(), placeholder));
                params.push(value.into_boxed());
            }
            QueryNode::Ne(field, value) => { /* AND field <> $n */ }
            QueryNode::In(field, values) => { /* AND field = ANY($n) */ }
            QueryNode::Between(field, lo, hi) => { /* AND field BETWEEN $n AND $n+1 */ }
            QueryNode::IsNull(field) => { /* AND field IS NULL */ }
            QueryNode::HasRelation(StudentRelation::Parent, inner) => {
                self.emit_parent_exists(inner, sql, params);
            }
            // every variant is matched; missing arms are compile errors
        }
    }
}
```

The `column()` method — mapping an enum variant to a physical column
name — is the **only** point in the adapter where the macro-emitted
type meets the SQL schema. No `String` field name, no
`serde_json::Value`, no schema introspection is permitted anywhere
else on the read path.

### Translating `HasRelation` into `WHERE EXISTS`

A `HasRelation(StudentRelation::Parent, Box<QueryNode<ParentField>>)`
node is a request to filter parent rows by an attribute of the
related entity. The PostgreSQL translation uses a correlated subquery
so the caller's closure body stays a clean predicate on the related
table:

```rust
fn emit_parent_exists(
    &self,
    inner: &QueryNode<ParentField>,
    sql: &mut String,
    params: &mut Vec<Box<dyn ToSql>>,
) {
    sql.push_str(
        " AND EXISTS (SELECT 1 FROM parents p WHERE p.id = students.parent_id"
    );
    self.emit_parent_node(inner, "p", sql, params);
    sql.push(')');
}
```

The same closure body can be translated to an `INNER JOIN` instead —
useful when the planner benefits from a hash join over a correlated
subquery. The choice is a storage-level decision; the caller's code
does not change:

```sql
SELECT s.*
FROM students s
INNER JOIN parents p ON p.id = s.parent_id
WHERE s.school_id = $1
  AND p.billing_status = $2
```

For MongoDB the adapter emits a `$lookup` followed by a `$match` on
the joined document. The closure body the caller wrote stays
untouched; only the dialect differs.

### Hydration via `.with(StudentRelation::Parent)`

The hydration set lives on `StudentQuery` as a
`BTreeSet<StudentRelation>`. The adapter must fully hydrate the
requested relations before returning. Two storage-level strategies are
available, both of which are implementation details hidden behind the
repository signature.

**1. JOIN.** When the relation is many-to-one or one-to-one, fold the
hydration into the primary query:

```sql
SELECT s.id, s.status, s.last_name, s.class_id, s.parent_id,
       p.id AS p_id, p.last_name AS p_last_name
FROM students s
LEFT JOIN parents p ON p.id = s.parent_id
WHERE s.school_id = $1
```

**2. Batched secondary load.** When the relation is one-to-many or
the join would explode the row set, fetch the parent ids from the
primary result, run a single `WHERE id = ANY($1)` against the related
table, and stitch in application code.

Either strategy is an implementation detail. The repository method
**must not** return a partial result. If the strategy fails — missing
index, transient error — the repository returns
`StorageError::HydrationFailure` and the engine maps it to
`DomainError::Infrastructure`. The application layer never sees a
half-hydrated aggregate. See `docs/ports/storage.md` "Hydration
Atomicity" for the port-level rules.

## Macro Boundary

The macro and the adapter are separated by a hard compile-time /
runtime boundary:

|                       | Macro (compile time)                                        | Adapter (runtime)                                |
| --------------------- | ----------------------------------------------------------- | ------------------------------------------------ |
| Runs when             | `cargo build`                                               | request handling                                 |
| Reads                 | struct source                                               | the AST the builder emitted                      |
| Emits                 | `StudentField`, `StudentQueryBuilder`, `StudentRelation`, `StudentQuery` | SQL, NoSQL, connection calls                     |
| Knows the database    | **No**                                                      | **Yes**                                          |
| Performs I/O          | **No**                                                      | **Yes**                                          |

The macro **must not** open connections, parse runtime configuration,
or emit any storage-specific syntax. The adapter **must not** import a
`StudentField` enum into a `where` clause on a hand-written type; the
enum only exists because the macro emitted it from the struct.

Adding a new queryable field is a single change to the struct (an
attribute). The macro regenerates the field enum; the adapter's
`match` fails to compile until a new arm is added. The same
exhaustiveness property applies to operators and relations: adding
`QueryNode::Between` requires a new translation arm or the build
fails.

## Outbox

The outbox is a table in the same database:

```sql
CREATE TABLE outbox (
    event_id UUID PRIMARY KEY,
    event_type TEXT NOT NULL,
    aggregate_id UUID NOT NULL,
    school_id INT NOT NULL,
    payload JSONB NOT NULL,
    occurred_at TIMESTAMPTZ NOT NULL,
    published_at TIMESTAMPTZ,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX ix_outbox_unpublished ON outbox (created_at) WHERE published_at IS NULL;
```

The outbox relay polls `WHERE published_at IS NULL` and publishes
events to the bus. On success it sets `published_at = NOW()`.

## Tenant Isolation

Every query MUST include `school_id = $N`. The adapter enforces this
through a wrapper:

```rust
fn with_school_filter(&self, sql: &mut String) {
    if !sql.contains("school_id =") {
        panic!("query missing school_id filter: {}", sql);
    }
}
```

A debug-mode assertion catches missing filters. Release builds rely
on the engine's compile-time guarantee.

Row-level security is recommended as defense in depth:

```sql
ALTER TABLE students ENABLE ROW LEVEL SECURITY;
CREATE POLICY students_school_isolation ON students
    USING (school_id = current_setting('app.current_school_id', true)::int);
```

The adapter sets `app.current_school_id` on every connection from
the `tenant.school_id`.

## SQLite Differences

SQLite has no row-level security. Tenant isolation is enforced solely
by the application code. Other differences:

- No `JSONB` — use `TEXT` and parse manually.
- No `UUID` — use `TEXT` storing the canonical UUID string.
- No concurrent writes — use `BEGIN IMMEDIATE`.
- No connection pool — use a Mutex around the connection.

## SurrealDB / MongoDB

Document stores are supported by translating the relational schema
into documents:

- Each aggregate root is a document.
- Owned children are embedded sub-documents.
- References are foreign keys stored as ids.
- The adapter translates the macro-emitted `QueryNode<StudentField>`
  AST into SurrealQL or MongoDB queries. The macro does not emit any
  storage-specific syntax.

The engine provides a `DocumentAdapter` trait that abstracts over
relational and document adapters.

## Testing

The storage adapter ships with:

- A unit test for every repository method.
- An integration test suite using testcontainers (PostgreSQL) or
  rusqlite (SQLite).
- A parity test verifying identical results across adapters.
- A tenancy test verifying cross-tenant denial.
- A failure-injection test (deadlock retry, connection drop).
- A load test (10k attendance marks in <5s on PostgreSQL).

## Worked Example

A consumer wires a PostgreSQL storage adapter:

```rust
let storage: Arc<dyn StorageAdapter> = Arc::new(
    PostgresStorage::connect(&env::var("DATABASE_URL")?).await?
);
storage.migrate().await?;  // run consumer migrations
let engine = Engine::builder().storage(storage).build().await?;
```

The consumer's migrations live in `migrations/` (e.g. using
`refinery` or `sqlx-migrate`). The engine's expected schema is
documented in `docs/specs/<domain>/repositories.md`.

## Performance Tips

- Use prepared statements (sqlx caches them).
- Batch inserts in transactions.
- Use `COPY` for bulk loads.
- Index every column in WHERE and ORDER BY.
- Avoid `SELECT *`. Select only the columns the aggregate needs.
- Use connection pooling (max ~20 connections per CPU).
- Monitor slow queries via `pg_stat_statements`.
