# ADR-006: Compile-Time-Safe Query Layer

## Status

Accepted.

## Context

The school domain has hundreds of aggregates. The consumer's
UI needs to ask questions like:

- "List active students in class 7B, sorted by last name,
  paginated."
- "Show me invoices where the due date is in the past and the
  balance is greater than zero."
- "Find all staff members whose leave request is pending."

These questions are domain-specific. They are not generic
"SQL where clauses"; they are "domain queries" expressed in
the engine's ubiquitous language.

The naive approach — letting the consumer write raw SQL or
pass string field names — is dangerous in a multi-tenant,
type-safe engine:

- A typo in a column name is a runtime error.
- A field that doesn't exist on the aggregate is a runtime
  error.
- A rename of a column is a search-and-replace exercise
  across every consumer.
- Cross-tenant leakage is one forgotten `WHERE school_id = ?`
  away.
- Subtle type mismatches (passing a `String` where an `i64`
  is expected) only surface at query time.

The "Eloquent-like" approach — chainable methods with
string-typed fields — is what most ORMs offer. It is
ergonomic but loses type safety.

## Decision

SMScore adopts a **compile-time-safe query layer** built on
Rust's type system.

Concretely:

1. **Every aggregate has a `*Field` enum** that enumerates
   its queryable fields. The enum is generated by a derive
   macro from the aggregate's struct definition.
2. **The query builder is generic over the aggregate and
   its field enum.** Method signatures are
   `where_eq(&mut self, field: StudentField, value: ...)`.
3. **Field values are strongly typed.** A `StudentField::Status`
   accepts a `StudentStatus`, not a `&str`. A
   `InvoiceField::Balance` accepts a `Money`, not an `f64`.
4. **The builder carries the active `TenantContext` from
   construction.** A `StudentQuery` cannot be built without
   one.
5. **The builder is a thin DSL**, not a fluent SQL generator.
   The engine translates the builder into the storage
   adapter's native query (SQL, SurrealQL, etc.).
6. **Repositories MAY expose domain-specific optimized
   queries** that bypass the builder for performance
   (e.g. `student_repository.find_active_for_class(class_id)`).
   The optimized methods are still strongly typed and still
   carry the `TenantContext`.
7. **No string field names, no `serde_json::Value`, no raw
   column names in consumer code.** A `where("first_name",
   ...)` is a compile error.
8. **Pagination is part of the builder**, with stable
   cursor or page-based semantics. A `Page` value object
   is the only accepted input.

The query layer is documented in `docs/query_layer.md` and
in the per-domain `repositories.md` files.

## Consequences

### Positive

- **Field-name typos are compile errors.** A consumer
  writing `StudentField::Statuz` fails to compile.
- **Type mismatches are compile errors.** A consumer
  passing a `String` for a `DateTime` field fails to
  compile.
- **Tenant binding is mandatory.** A query cannot be
  built without a `TenantContext`.
- **The query layer is its own documentation.** The
  `StudentField` enum is the list of queryable fields;
  reading it tells the consumer what they can ask.
- **Refactoring is safe.** Renaming a field ripples
  through the type system; consumers are forced to
  update.
- **No SQL injection.** The builder produces parameterized
  queries; the consumer never assembles a string.
- **Domain-specific optimized queries are still
  type-safe.** A repository method
  `find_active_for_class(class_id)` is checked at
  compile time; the consumer cannot pass the wrong type.

### Negative

- **Boilerplate per aggregate.** Each aggregate has a
  `*Field` enum. The derive macro reduces but does not
  eliminate the boilerplate.
- **The builder is verbose.** A query that would be one
  line of Eloquent is five lines in the engine. The
  tradeoff is intentional.
- **Domain-specific queries are sometimes faster than
  the builder.** The builder is generic; an optimized
  repository method can use a covering index. The
  builder is the floor, not the ceiling.
- **Consumers cannot write ad-hoc SQL.** For edge cases
  (e.g. complex reports), the consumer uses the
  `Report.Generate` command, which is itself capability-
  gated.

### Mitigations

- A derive macro `#[derive(QueryFields)]` generates the
  field enum from a struct.
- A builder trait provides common methods
  (`where_eq`, `where_in`, `order_by`, `limit`, `paginate`)
  that work for any aggregate.
- An `IndexHint` mechanism lets the consumer (or the
  storage adapter) declare which index a query should
  use, without changing the type.
- The `Report.Generate` command handles ad-hoc analytics
  in a typed, capability-gated way.

## Alternatives Considered

### 1. Raw SQL with parameter binding

The consumer writes SQL. Rejected because it bypasses the
type system, makes refactoring hard, and exposes the
engine to SQL injection.

### 2. Eloquent-like string-typed field names

The consumer writes `query.where("first_name", "Anita")`.
Rejected because typos are runtime errors and refactoring
is unsafe.

### 3. ORM with `#[derive(Queryable)]` and reflection

The consumer marks a struct queryable; the ORM infers
fields at runtime. Rejected because the engine's code
standards forbid runtime reflection.

### 4. GraphQL

The consumer writes a GraphQL query. Rejected because
GraphQL is a wire format, not a query language that
maps to the engine's aggregates. The consumer can
expose a GraphQL API on top of the engine's query
layer; the engine itself is not GraphQL.

### 5. No query layer; consumer reads from raw tables

The consumer writes its own SQL. Rejected because it
bypasses every invariant the engine has.

### 6. Stored procedures

Database-stored domain logic. Rejected because the engine
does not own a specific database, and because business
rules belong in Rust, not in DDL.
