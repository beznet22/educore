# Payment Port

## Purpose

The payment port processes payments for invoices, fees, payroll, and
other monetary obligations. The engine does not own card processing,
gateway integrations, or reconciliation logic. The consumer supplies an
adapter.

## Trait: `PaymentProvider`

```rust
#[async_trait]
pub trait PaymentProvider: Send + Sync + std::fmt::Debug {
    async fn charge(&self, request: ChargeRequest) -> Result<PaymentReceipt>;
    async fn refund(&self, request: RefundRequest) -> Result<RefundReceipt>;
    async fn status(&self, payment_id: PaymentId) -> Result<PaymentStatus>;
    async fn list_methods(&self, tenant: TenantContext) -> Result<Vec<PaymentMethodInfo>>;
}
```

The trait is object-safe.

## ChargeRequest

```rust
pub struct ChargeRequest {
    pub tenant: TenantContext,
    pub amount: Money,
    pub method: PaymentMethod,
    pub invoice_ids: Vec<InvoiceId>,
    pub customer: CustomerRef,
    pub description: String,
    pub metadata: BTreeMap<String, String>,
    pub idempotency_key: IdempotencyKey,
    pub correlation_id: CorrelationId,
    pub capture: bool,  // false = authorize only, true = capture
    pub webhook_url: Option<Url>,
}
```

`capture = false` is used for two-step flows (auth-then-capture) such
as school fee card payments that need approval.

## PaymentMethod

```rust
pub enum PaymentMethod {
    Cash,
    Cheque { number: String, bank: String, date: NaiveDate },
    BankTransfer { reference: String, bank_account: BankAccountId },
    Card { token: CardToken, save: bool },
    Gateway { gateway: GatewayName, token: GatewayToken, return_url: Url },
    Wallet { wallet_id: WalletId, pin: SecretString },
    ManualAdjustment { reason: String, approver: UserId },
}
```

`Card` and `Gateway` methods do not carry raw card numbers. The
consumer's frontend tokenizes the card via the gateway's SDK and passes
a token. The engine never sees the PAN.

## PaymentReceipt

```rust
pub struct PaymentReceipt {
    pub payment_id: PaymentId,
    pub provider_payment_id: Option<String>,
    pub status: PaymentStatus,
    pub amount: Money,
    pub method: PaymentMethodKind,
    pub authorized_at: Option<Timestamp>,
    pub captured_at: Option<Timestamp>,
    pub fees: Vec<PaymentFee>,
    pub net: Money,
    pub receipt_url: Option<Url>,
    pub metadata: BTreeMap<String, String>,
}
```

The receipt is durable. The engine stores it and emits a
`PaymentReceived` event.

## PaymentStatus

```rust
pub enum PaymentStatus {
    Pending,
    Authorized { auth_code: String, expires_at: Timestamp },
    Captured { at: Timestamp },
    Failed { reason: String, code: Option<String> },
    Refunded { amount: Money, at: Timestamp, reason: String },
    PartiallyRefunded { refunded: Money, remaining: Money },
    Disputed { dispute_id: String, reason: String, opened_at: Timestamp },
    Cancelled { at: Timestamp, reason: String },
}
```

## Refund

```rust
pub struct RefundRequest {
    pub tenant: TenantContext,
    pub original_payment_id: PaymentId,
    pub amount: Money,                 // can be partial
    pub reason: String,
    pub refund_to: RefundDestination,
    pub idempotency_key: IdempotencyKey,
}

pub enum RefundDestination {
    OriginalMethod,
    Wallet { wallet_id: WalletId },
    BankAccount { bank_account_id: BankAccountId },
    CreditNote,                        // a future invoice
}
```

The engine issues refunds only via commands that require an
`Finance.Refund` capability. The adapter is responsible for the
actual money movement.

## Bank Slip Flow (Offline)

Schools that collect payments via bank deposit (slip) use an offline
flow:

1. School admin records a bank slip with reference number, amount,
   date, and a photo of the slip.
2. The slip is stored in the file storage port.
3. A `BankPaymentSlipGenerated` event is emitted.
4. A school approver (or accountant) reviews the slip.
5. On approval, a `BankPaymentApproved` event is emitted, which
   triggers the payment record.

The payment port handles the slip generation and approval workflow
through commands (`GenerateBankPaymentSlip`, `ApproveBankPayment`).
The provider adapter is not involved in the offline slip flow; the
engine handles the state transitions directly.

## Webhook Flow (Online)

For online gateways (Stripe, etc.), the consumer application exposes
a webhook endpoint that calls the payment port's `status(payment_id)`.
The engine correlates the webhook to the originating command via
`idempotency_key` and updates the payment status.

The engine does not own the webhook endpoint; the consumer registers
the URL with the gateway.

## Idempotency

`IdempotencyKey` is generated by the engine and persisted. The adapter
uses it to deduplicate retries. A retry of the same charge returns the
same `PaymentReceipt` without re-charging.

## Multi-Currency

`Money` carries a `CurrencyCode`. The adapter rejects a charge whose
currency does not match the invoice. Cross-currency is not supported
in v1; the consumer is responsible for FX conversion before charging.

## Settlement & Reconciliation

The adapter is responsible for settlement reporting (when does the
money land in the school's bank account?). The engine consumes
settlement events:

```rust
#[async_trait]
pub trait PaymentProvider {
    async fn settlement(&self, request: SettlementRequest) -> Result<Settlement>;
}
```

A `Settlement` is a batch of captured payments that have settled into
the school's bank account. The engine matches settlement lines to
`PaymentReceipt` records by `provider_payment_id`.

## Error Type

```rust
#[derive(Debug, thiserror::Error)]
pub enum PaymentError {
    #[error("invalid amount: {0}")] InvalidAmount(String),
    #[error("currency mismatch: invoice={invoice}, payment={payment}")]
    CurrencyMismatch { invoice: CurrencyCode, payment: CurrencyCode },
    #[error("card declined: {0}")] Declined(String),
    #[error("insufficient funds")] InsufficientFunds,
    #[error("3DS required")] ThreeDSRequired,
    #[error("rate limited")] RateLimited,
    #[error("provider error: {0}")] Provider(String),
    #[error("already refunded")] AlreadyRefunded,
    #[error("refund exceeds original")] RefundExceedsOriginal,
    #[error("infrastructure error: {0}")] Infrastructure(#[source] Box<dyn std::error::Error + Send + Sync>),
}
```

## Worked Example

```rust
let receipt = engine.payments().charge(ChargeRequest {
    tenant,
    amount: Money::new(dec!(1500.00), CurrencyCode::USD),
    method: PaymentMethod::Cash,
    invoice_ids: vec![invoice_id],
    customer: CustomerRef::Student(student_id),
    description: "Term 1 fees",
    metadata: Default::default(),
    idempotency_key,
    correlation_id,
    capture: true,
    webhook_url: None,
}).await?;

engine.fees().record_payment(RecordPaymentCommand {
    tenant,
    payment_id: receipt.payment_id,
    invoice_id,
    amount: receipt.amount,
    method: PaymentMethodKind::Cash,
}).await?;
```

## Offline Mode

In offline mode, the consumer uses the `Cash`, `Cheque`, `BankSlip`,
or `Wallet` methods. Online gateway methods are unavailable. The
engine still emits `PaymentReceived` events with status `Captured` and
a generated `provider_payment_id` of `local://cash/<uuid>`. On
reconnect, payments are reconciled with the central ledger.

## Audit

Every charge, refund, status change, and settlement is recorded in
the audit log with amount, currency, method, parties, and metadata.
Card data is never logged.
