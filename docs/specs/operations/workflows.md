# Operations Domain — Workflows

Workflows orchestrate commands, queries, and policies to fulfill a
business goal. They are documented as ordered, conditional steps.

## Backup Lifecycle Workflow

```text
1. SchoolAdmin (or scheduled job) issues CreateBackupCommand.
2. The system produces the backup file via the file-storage port.
3. The system writes the Backup row and emits BackupCreated.
4. SchoolAdmin can list backups and download them.
5. SchoolAdmin issues DeleteBackupCommand to remove an old
   backup.
6. SchoolAdmin issues RestoreBackupCommand to restore from a
   backup.
7. The system emits BackupRestored.
8. Platform subscribes and invalidates its in-memory caches.
```

**Pre-conditions:**

- The school has free storage quota (consumer-enforced).
- No other restore is in progress for the school.

**Failure paths:**

- File-storage failure → `InfrastructureError::StorageUnavailable`.
- Concurrent restore → `ConflictError::RestoreInProgress`.

## Job Scheduling Workflow

```text
1. A domain emits a job-required event.
2. A subscriber (in the same domain) issues ScheduleJobCommand
   with a serialized command envelope as the payload.
3. The Job row is written with `available_at = now`.
4. A job runner (port) dequeues the Job row, marks it reserved,
   and executes the payload.
5. On success, the runner issues MarkJobCompletedCommand; the
   Job row is deleted.
6. On failure, the runner increments `attempts`; if the retry
   budget is exhausted, the runner issues MarkJobFailedCommand.
7. The operations domain records a FailedJob row and emits
   FailedJobRecorded.
8. An operator may issue RetryFailedJobCommand to re-queue the
   job; a new Job row is created with the original payload.
```

## System Version Bump Workflow

```text
1. Build pipeline registers the new version (RegisterSystemVersion).
2. Build pipeline records the version history
   (RecordVersionHistory).
3. The operations domain derives SystemVersionBumped.
4. Settings subscribes and updates GeneralSettings.system_version.
5. Platform subscribes and refreshes the system-version display.
6. The new version is now visible across the engine.
```

## User Log Retention Workflow

```text
1. A user logs in.
2. The platform domain issues RecordUserLogCommand with the
   request context.
3. The UserLog row is written.
4. A nightly job (port) partitions the log by month.
5. A nightly job (port) purges UserLog rows older than the
   school's retention policy (default 365 days).
6. The purge is logged as a DeleteUserLog event for compliance.
```

## Maintenance Mode Workflow

```text
1. SchoolAdmin issues ConfigureMaintenanceCommand with the
   title, sub-title, and applicability.
2. SchoolAdmin issues EnableMaintenanceCommand.
3. The system writes maintenance_mode=true and emits
   MaintenanceEnabled.
4. The platform domain's authentication flow begins rejecting
   non-admin logins for the school.
5. Existing sessions are terminated (port-driven).
6. Operators perform the maintenance work.
7. SchoolAdmin issues DisableMaintenanceCommand.
8. The system writes maintenance_mode=false and emits
   MaintenanceDisabled.
9. The platform domain resumes normal authentication flow.
```

## Sidebar Layout Workflow

```text
1. SchoolAdmin creates a sidebar entry (CreateSidebarEntry).
2. SchoolAdmin reorders (ReorderSidebar).
3. The sidebar is rendered for users in the role.
4. To remove an entry, SchoolAdmin issues DeleteSidebarEntry.
```

## OAuth Token Refresh Workflow (Port-Driven, Documented)

```text
1. Consumer calls AuthProvider::refresh with the refresh token.
2. AuthProvider validates the refresh token and issues a new
   access token (and optionally a new refresh token).
3. The old access token is revoked.
4. The event log receives a token-issued event (operator-defined).
```

The engine does not own the OAuth tables; the consumer's
AuthProvider implementation does. The operations domain documents
the workflow for completeness.

## Password Reset Workflow (Port-Driven, Documented)

```text
1. User requests a password reset (RequestPasswordReset).
2. AuthProvider writes a row to password_resets with a hashed
   token.
3. AuthProvider emails the user a reset link.
4. User clicks the link and submits a new password
   (CompletePasswordReset).
5. AuthProvider validates the token, updates the user's password
   hash, and deletes the password_resets row.
6. The user is notified.
```

## Idempotency

- `CreateBackup` is idempotent on `(school_id, file_name)`. A
  duplicate returns the existing backup.
- `RestoreBackup` is not idempotent; a second call restores the
  same backup again.
- `ScheduleJob` is not idempotent by default. The job id is
  generated by the engine; duplicates are not produced.
- `RecordUserLog` is append-only and never idempotent.
- `EnableMaintenance` is idempotent on `school_id`.
