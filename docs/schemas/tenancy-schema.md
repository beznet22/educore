# Tenancy Schema

This document is **normative**. It defines how SMScore models
multi-tenancy, how `SchoolId` flows through every layer, and the
rules for cross-tenant operations.

## 1. The Tenant is a `School`

The unit of tenancy in SMScore is a `School`. Every aggregate belongs
to exactly one school. A school is the legal and operational boundary
for data, configuration, users, and authorization.

```text
School
├── Users (staff, students, parents)
├── Academic structures (classes, sections, subjects)
├── Finance (fees, payments, banking)
├── Roles and capabilities
├── Settings
├── Audit log
├── Backups
└── ... (every other aggregate)
```

Two schools in the same deployment have **no shared data**, **no
shared identifiers**, and **no shared configuration** unless an
explicit cross-tenant command is issued.

## 2. `SchoolId`

A `SchoolId` is a typed identifier:

```rust
pub struct SchoolId(Uuid);
```

Properties:

- It is a UUIDv7 (time-ordered) globally unique identifier.
- It is generated by the `IdGenerator` port at school creation time.
- It is **immutable**. A `SchoolId` never changes for the lifetime of
  the school. Renaming the school does not change its id.
- It is **opaque** to consumers. Strings are never parsed.

A `SchoolId` participates in **every** aggregate's primary key
construction:

```rust
pub struct StudentId(SchoolId, Uuid);
pub struct InvoiceId(SchoolId, Uuid);
pub struct PayrollId(SchoolId, Uuid);
```

The presence of `SchoolId` in the typed identifier makes cross-school
references a **compile-time** category error. Code that passes a
`StudentId` from school A into a function expecting school B's
`StudentId` will not typecheck.

## 3. `TenantContext`

The `TenantContext` is the active tenant for a single command or
query. It carries:

```text
TenantContext {
    school_id:       SchoolId,        // the active school
    actor_id:        UserId,          // the user (or SYSTEM) acting
    session_id:      Option<SessionId>,
    correlation_id:  CorrelationId,    // propagated to every event
    causation_id:    Option<EventId>,  // the event that caused this command
    user_type:       UserType,        // student, parent, teacher, etc.
    locale:          Locale,
    timezone:        TimeZone,
}
```

Rules:

- The `TenantContext` is **immutable** for the lifetime of a single
  command.
- It is constructed at the engine boundary (the consumer's
  authentication layer).
- It is **never** constructed by domain code. Domain code receives it
  as input and reads `school_id` from it.
- For background jobs, `actor_id` is the job's service user; the
  `correlation_id` is the job's run id.

## 4. Every Aggregate Anchors to a School

Every aggregate root carries a `SchoolId` as part of its identifier
and as a column in its primary table. The engine's storage adapters
MUST:

- Add a `school_id` column on every aggregate table.
- Add a row-level security policy on every aggregate table that
  filters by `school_id`.
- Inject a `school_id` predicate into every query unless the caller
  is an explicit, capability-gated cross-tenant operation.
- Reject writes whose `school_id` does not match the caller's
  `TenantContext::school_id`.

The engine's domain code is **not** responsible for re-checking the
`school_id` filter on every read; the storage adapter enforces it.
This is documented in `database-schema.md` § 3.

## 5. Cross-Tenant Operations

Cross-tenant operations are **never** the default. They require:

1. The actor holds the `Platform.CrossTenant` capability.
2. The actor's `user_type` is `SuperAdmin` (anchored to
   `SchoolId::PLATFORM`).
3. The command is explicitly a cross-tenant command
   (e.g. `Platform.School.Suspend`).
4. The audit record carries `cross_tenant = true` and lists the
   source and target schools.

Cross-tenant operations are limited to a small catalog:

- `Platform.School.Transfer` — move a student or staff from school A
  to school B.
- `Platform.School.Merge` — merge school B into school A.
- `Platform.School.Suspend` / `Unsuspend` — pause / resume a school.
- `Platform.School.Clone` — copy configuration from one school to
  another (no PII).

The engine does **not** allow ad-hoc cross-tenant queries. If a
consumer needs aggregated data across schools, it must use the
reporting port, which is itself an explicit, capability-gated
operation.

## 6. Tenant Isolation in Queries

The query layer enforces tenant isolation at multiple levels:

1. **Type system** — `StudentQuery` requires a `SchoolId` to
   construct.
2. **Builder** — every query method takes a `&TenantContext`. The
   builder carries it.
3. **Storage adapter** — the adapter injects the `school_id` filter
   into the generated SQL/Query.
4. **Database** — row-level security policies reject any result the
   adapter did not constrain.

A query cannot accidentally return rows from another school. The
compile-time types prevent passing a wrong `SchoolId`; the storage
adapter prevents running unfiltered queries; the database provides
the last line of defense.

## 7. Tenant-Scoped Identifiers

Some identifiers are tenant-scoped (unique within a school) rather
than globally unique. The engine treats them as such:

- `Student::admission_number` — unique within `(school_id,
  admission_number)`.
- `Student::roll_number` — unique within `(school_id, class_id,
  section_id, academic_year_id, roll_number)`.
- `Invoice::invoice_number` — unique within `(school_id, prefix,
  start_form)`.
- `User::email` — unique within `(school_id, lower(email))`.
- `User::phone` — unique within `(school_id, normalized_phone)`.
- `Subject::code` — unique within `(school_id, code)`.

Globally-unique identifiers (`StudentId`, `InvoiceId`, `UserId`) are
the canonical primary keys. Tenant-scoped identifiers are alternate
keys (unique constraints), not primary keys.

## 8. Multi-Tenant SaaS vs Single-Tenant

The engine supports both deployment modes without code change.

### 8.1 Multi-Tenant SaaS

- Many schools in one database, discriminated by `school_id`.
- Row-level security policies are mandatory.
- A super-admin role anchored to `SchoolId::PLATFORM` exists.
- The engine's storage adapter configures a per-session
  `app.school_id` and the database policies enforce isolation.

### 8.2 Single-Tenant On-Premise

- One school per database.
- The `school_id` column contains a single value (often `1`).
- The engine treats this case identically to SaaS — the same
  `school_id` filter is applied, but it matches every row.
- No `SuperAdmin` cross-tenant role is needed.

### 8.3 Multi-Database

A consumer MAY run a separate database per school even in SaaS mode.
The engine's storage adapter is initialized with a per-school
connection. The engine itself does not care; the adapter translates
the `SchoolId` into a connection lookup.

## 9. Per-Tenant Configuration

Each school has its own configuration values, anchored to `SchoolId`:

- **Settings** (theme, language, currency, timezone, academic year
  format).
- **Capabilities** (roles, capability assignments).
- **Modules** (which optional modules are enabled — academic-only,
  finance-only, etc.).
- **Payment gateways** (which providers are configured for the
  school).
- **Notification providers** (which channels are active).

Per-tenant configuration is read by every domain. A domain's command
reads its configuration from the active `SchoolId` and applies it
uniformly. There is no global "default school" configuration.

The engine provides a `ConfigurationService` port. The consumer
implements a backend (database, file, KV store). All reads are
tenant-scoped.

## 10. Tenant Onboarding

A new school is created via the `Platform.School.Onboard` command.
The command is multi-step and capability-gated:

1. **Provision** — generate `SchoolId`, write the `School` aggregate
   with the school name, code, address, contact, package, and
   onboarding metadata.
2. **Seed roles** — install the default `Role` catalog
   (SuperAdmin-bound role for the new school admin).
3. **Seed settings** — install the default `GeneralSettings` row.
4. **Seed academic structures** — install the default `AcademicYear`
   and any classes the school opts into.
5. **Assign admin** — bind the school's first admin user.
6. **Emit** `Platform.School.Onboarded` event.

Each step is its own command and is independently auditable. A
failed step does not roll back the school; the onboarding workflow
records its progress and resumes from the next step.

## 11. Tenant Deletion Policy

A school is deleted via `Platform.School.Delete`. The deletion is:

1. **Soft** by default — `active_status = 0`, the school becomes
   invisible to non-platform admins.
2. **Hard** after a confirmation window (default 30 days) — every
   row in every aggregate table for the school is purged.
3. **Anonymized** for any PII required to be retained for compliance
   (e.g. financial records for 7 years). The audit log retains a
   redacted summary identifying the school by a one-way hash.

The engine does not auto-purge. The consumer's operations team
reviews and confirms.

## 12. Tenant Transfer

A `Platform.School.Transfer` command moves a school from one
deployment to another. The flow:

1. **Snapshot** — every aggregate for the school is exported as a
   self-contained, versioned, signed bundle.
2. **Purge** — the source deployment purges the school (with a
   retention window).
3. **Import** — the target deployment imports the bundle into a new
   `SchoolId`.
4. **Re-bind** — external integrations (payment webhooks, email
   routes) are re-pointed to the target.

The bundle format is a port. Default is a tarball with a manifest and
per-aggregate JSONL files.

## 13. Tenant Lifecycle Events

The platform domain emits the following events, all carrying
`school_id`:

- `Platform.School.Onboarded` — a new school is live.
- `Platform.School.Suspended` — admin has paused the school.
- `Platform.School.Reinstated` — admin has resumed the school.
- `Platform.School.Configured` — bulk configuration import applied.
- `Platform.School.Transferred` — school moved to another deployment.
- `Platform.School.Deleted` — soft delete applied.
- `Platform.School.Purged` — hard delete applied (after retention).

Subscribers:

- **Finance** — closes open ledgers on suspension, archives on
  transfer.
- **Communication** — stops notifications on suspension, restarts on
  reinstatement.
- **Audit** — records every state change with the actor and the
  reason.

## 14. Per-Tenant Feature Flags

Per-tenant feature flags are values stored in the configuration
service. They are keyed by `(school_id, flag_name)`. Flags are
typed booleans / strings / numbers and are read at command dispatch
time. The engine does not interpret flag values; domains and
consumers do.

## 15. Multi-Tenant Query Boundaries

The query layer enforces three boundaries:

1. **Per-school query** — the default. The query's result set is
   bounded to the active `school_id`.
2. **Aggregate query** — within a school, the query is bounded to
   one aggregate type.
3. **Cross-aggregate read** — a query that joins across aggregates
   stays within the active school.

A cross-school query is **not** exposed. Reporting across schools
uses the `Report.Generate` command with the `Platform.CrossTenant`
capability; the result is a single read-only snapshot, not a
re-runnable query.

## 16. The Bootstrap School

The engine ships with a bootstrap school (id `PLATFORM_BOOTSTRAP`)
that owns the `SuperAdmin` role and the global lookup data
(`Country`, `Currency`, `TimeZone`, `Language`). The bootstrap school
is created at engine initialization and cannot be deleted. It is
the parent of the global registry and is invisible to all other
schools.
