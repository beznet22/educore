# SQL Dialects — Index

The engine guarantees **table parity**: the same table name and
column semantics in MySQL, SQLite, and PostgreSQL. The dialect
files in this folder document the per-dialect DDL conventions for
the engine's storage adapter authors.

## Audience

The reader of this folder is the **engine crate author** — the
implementer of `smsengine-storage-mysql`, `smsengine-storage-sqlite`,
or `smsengine-storage-postgres`. The consumer of those crates
(whoever calls `MysqlStorage::builder().url(env::var("DATABASE_URL")?).build()`)
does not read these files; they read `docs/ports/storage.md` and the
adapter's README.

The crate author reads these files to know:

- Which SQL types to use for `CHAR(36)`, `BINARY(16)`, `JSON`,
  `TIMESTAMP` in their backend.
- How to emit DDL for the engine's invariants on a new aggregate.
- How to translate the `database-schema.md` rules into the
  backend's native syntax.
- What is and is not supported by each backend (e.g. PG has native
  `UUID`; SQLite does not; MySQL needs a feature flag for
  `CHECK` constraints pre-8.0.16).

## Files

| File | Purpose |
| --- | --- |
| `mysql.md` | MySQL 8+ DDL conventions (InnoDB, `utf8mb4_unicode_ci`, identifier quoting with backticks, `JSON` type, `ROW LEVEL SECURITY` since 8.0) |
| `sqlite.md` | SQLite 3.x DDL conventions (no charset, no RLS, dynamic typing, `WITHOUT ROWID`, JSON as `TEXT`, identifier quoting with double quotes or backticks) |
| `postgresql.md` | PostgreSQL 14+ DDL conventions (native `UUID` and `JSONB` types, `CREATE POLICY` for RLS, schemas-as-namespaces, identifier quoting with double quotes, `BIGINT GENERATED BY DEFAULT AS IDENTITY`) |
| `comparison.md` | feature-by-feature side-by-side table — for every engine feature, which backend supports it natively, which needs an adapter workaround, which does not support it at all |

## What the engine contract is

The engine's storage port (`StorageAdapter` per
`docs/ports/storage.md`) is the abstract interface. Each adapter
implements it for one backend. The DDL emitted by the adapter
conforms to the conventions in this folder.

The engine's invariants (per `docs/schemas/database-schema.md`):

- **Identifier parity**: `outbox` is `outbox` in every backend.
- **Type parity**: every column has a type, nullable, default. The
  `CHAR(36)` for ids is the same character set (utf8mb4 in MySQL,
  utf8 in PG, utf8 in SQLite) and the same byte length (36 chars).
- **Engine invariants**: every aggregate has the 7 invariant
  columns. The adapter's DDL emits them.
- **Constraint parity**: every aggregate has the
  `(school_id, active_status)` index. The adapter's DDL emits it.
- **Tenant isolation**: every aggregate's `school_id` is `NOT NULL`.
  The adapter's DDL emits `NOT NULL` and the row-level security
  policy where supported.

## What the engine contract is NOT

- **No** per-backend table renames. The table name is the same.
- **No** per-backend column type flexibility beyond the dialect
  file's mapping (e.g. `CHAR(36)` becomes `TEXT` in SQLite; this is
  documented and is the only acceptable adaptation).
- **No** schema-as-namespace in MySQL or SQLite. PostgreSQL's schemas
  are optional and consumer-driven; the engine does not require
  them.

## DDL example: the `outbox` table in all three backends

The canonical form is in `migrations/engine/0000_engine_core.mysql.sql`
(MySQL). For PG and SQLite, see
`migrations/engine/0000_engine_core.postgres.sql` and
`migrations/engine/0000_engine_core.sqlite.sql`. The adapter's
`create_outbox_ddl()` method emits the dialect-specific form at
runtime, matching the canonical file for the same backend. See
the per-dialect files for the exact DDL.

## How to extend

When the engine adds a new cross-cutting table (e.g. a new port
implementation needs a new engine-internal table):

1. Add the MySQL DDL to `migrations/engine/0000_engine_core.mysql.sql`
   (the canonical MySQL reference).
2. Add the PG DDL to `migrations/engine/0000_engine_core.postgres.sql`
   (translate the MySQL form to PG syntax: `UUID`, `JSONB`, etc.).
3. Add the SQLite DDL to `migrations/engine/0000_engine_core.sqlite.sql`
   (translate to SQLite syntax: `TEXT` for ids, no RLS, etc.).
4. Add a comparison row to `docs/schemas/sql-dialects/comparison.md`.
5. Update the doc reference in this README's "Runtime DDL emission"
   section.
6. Implement `create_<table>_ddl()` in each adapter crate, returning
   the dialect-specific DDL string.
7. Add a smoke test in each adapter crate that runs the DDL against
   an in-memory database and verifies the engine's invariants.

## Runtime DDL emission — end-to-end flow

The engine's schema is not "a set of `.sql` migration files applied
by an external migration runner." It is a typed domain layer that
emits DDL at **schema-creation time** (once per process lifetime,
not per query). The flow has five steps:

### Step 1 — Design contract (`docs/specs/<domain>/tables.md`)

The human-readable design doc. A row like
`| `academic_students` | Student | The student record |` is the
**design intent** — what the architect agreed to. It is not
executable.

### Step 2 — Type contract (`crates/<domain>/src/aggregate.rs`)

```rust
pub struct Student {
    pub id: StudentId,
    pub school_id: SchoolId,
    pub admission_number: AdmissionNumber,
    pub first_name: PersonName,
    pub last_name: PersonName,
    pub date_of_birth: NaiveDate,
    pub active_status: ActiveStatus,
    pub created_at: Timestamp,
    pub updated_at: Timestamp,
    pub created_by: UserId,
    pub updated_by: UserId,
    pub version: Version,
    pub etag: Etag,
    pub last_event_id: Option<EventId>,
    pub correlation_id: Option<CorrelationId>,
    pub source: Option<Source>,
}
```

The Rust compiler enforces the type contract. It is the
**authoritative source of truth** at the type level.

### Step 3 — Machine contract (`crates/<domain>/src/entities.rs`)

A `#[derive(DomainQuery)]` macro reads the field list and emits an
in-memory AST (`EntityDescriptor`):

```rust
#[derive(DomainQuery)]
#[domain_query(table = "academic_students", aggregate = "Student")]
pub struct StudentRow {
    pub id: UuidV7,
    pub school_id: UuidV7,
    // ... one field per column ...
}
```

The macro output is **dialect-agnostic** — it carries table name,
column types, nullable, defaults, indexes, FKs, RLS policies as a
typed Rust data structure. No SQL strings.

### Step 4 — Adapter emission (`smsengine-storage-<db>`)

The storage adapter walks the AST and emits the dialect-specific
DDL string. For the 6 engine cross-cutting tables (outbox,
audit_log, idempotency, event_log, schema_registry, system_user),
the adapter `include_str!`s the canonical `.sql` file at compile
time:

The 3 canonical dialect DDL files in `migrations/engine/` are the
source of truth. The 6 cross-cutting tables are hard-coded in the
adapter via `include_str!`. The ~310 domain tables are macro-emitted
from the AST. There is no per-table `.sql` file for domain tables —
they are generated from the typed Rust struct at runtime.

```rust
// crates/smsengine-storage-postgres/src/lib.rs
const ENGINE_CORE_DDLS: &[&str] = &[
    include_str!("../../migrations/engine/0000_engine_core.postgres.sql"),
];
```

For the ~310 domain tables, the adapter walks the macro-emitted
AST and renders each table's DDL string at runtime. The 6
cross-cutting tables are hard-coded; the ~310 domain tables are
macro-driven. The two paths are mutually exclusive — no table is
emitted twice.

### Step 5 — Consumer startup

```rust
// consumer's main.rs
storage.create_schema().await?;
```

The adapter iterates all tables (6 hard-coded + ~310 macro-emitted)
and runs each DDL string against the connection pool. The
`create_schema()` call is made **once per process lifetime** — at
startup, or via a dedicated migration tool. It is not on the
query hot path.

### The five views of the same thing

| View | Lives in | Read by | Format | When |
| --- | --- | --- | --- | --- |
| **Design contract** | `docs/specs/<domain>/tables.md` | Humans (architects, reviewers) | Markdown table | Pre-implementation |
| **Type contract** | `crates/<domain>/src/aggregate.rs` | Rust compiler + macro | Rust struct | Compile time |
| **Machine contract** | `crates/<domain>/src/entities.rs` (macro output) | Storage adapter | In-memory typed AST | Compile time (macro) / runtime (in-memory) |
| **Canonical SQL** | `migrations/engine/0000_engine_core.<dialect>.sql` | DBA / engine author / `include_str!` | `.sql` file | Compile time (via `include_str!`) |
| **Executable DDL** | Adapter-emitted string at runtime | Database engine | SQL string | Schema-creation time only |

The 15 `migrations/0001_*.sql`–`0015_*.sql` files are the **legacy
Schoolify/InfixEdu dump** — research source only, not engine
schema. They are not read by the engine at runtime. They are read
by the data-migration plan (`docs/schemas/data-migration/`) and
the graphify plugin (`schoolify/graphify-out/`).

### Why not 315 `.sql` files?

The ~310 domain tables are emitted from the macro AST at runtime
in the adapter, not from `.sql` files. The trade-off:

- **6 cross-cutting tables**: hard-coded `.sql` in
  `migrations/engine/` (small, fixed, engine-internal). Adapter
  uses `include_str!`.
- **~310 domain tables**: macro-emitted AST → adapter-emitted
  string at runtime. No per-table file.

This keeps the file count manageable (~3 cross-cutting files vs.
~315 domain files) and ensures the engine invariants propagate
automatically when the AST changes. The `include_str!` pattern
guarantees no drift between the canonical file and the runtime
constant.

### Performance: is "runtime emission" slow?

No.

- **What the adapter does at `create_schema()`**: walks the AST
  (~310 tables × ~20 columns = ~6,200 field descriptors), builds
  a `String` per table (~200–500 bytes), runs the DDL via
  `sqlx::query(&ddl).execute(&pool)`.
- **String build time**: <10 ms total for ~310 tables. Trivial.
- **DB round-trip time**: dominates. Each `CREATE TABLE` +
  `CREATE INDEX` × 310 × ~3 statements = ~930 statements × ~20 ms
  = ~6 s for MySQL, faster for SQLite. The DB waits on the
  network, not on the string concat.
- **Hot-path cost**: zero. `create_schema()` is called once per
  process lifetime. Queries use the pre-existing schema.

The string is emitted (built from a typed AST), not parsed. The
DB engine parses the emitted string; the adapter does not.

### Why not compile-time emission via `build.rs`?

A `build.rs` could walk the macro AST and write `.sql` files at
`cargo build` time, which the adapter would then `include_str!`.
This adds a build step (slower `cargo build`) and generated-file
noise to source control, in exchange for a guarantee that the
DDL is final before the binary runs. The benefit is small; the
cost is real. The runtime-emission approach is preferred.
